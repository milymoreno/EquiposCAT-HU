# HU-IE-15-03: Creación Automática y Unificación de Proveedores en SII 2.0

**Proceso:** IMPORTACIÓN DE EQUIPOS CAT  
**Subproceso:** GESTIÓN DE PROVEEDORES / EXPORTADORES  
**Requerimiento Original:** REQ-15 — El sistema debe migrar el código de proveedor de 3 dígitos (SII) al código de 6 dígitos (D365 - Vendor Account), eliminando la tabla de mapeo manual. Los proveedores deben crearse automáticamente en el SII a partir de la información de D365.  
**Versión:** 2.0  
**Fecha de revisión:** 2026-03-05  
**Estado:** Revisada con base en sesión de levantamiento (02-Mar-2026)

---

## Historia de Usuario

### Como:
Sistema SII 2.0 / Analista de Comercio Exterior

### Quiero:
Que el SII 2.0 reciba y gestione proveedores/exportadores directamente usando el código D365 (6 dígitos), creándolos automáticamente si no existen al sincronizar facturas, y permitiendo completar o modificar sus datos directamente en el SII.

### Para:
Eliminar la dependencia de la tabla manual de mapeo SII ↔ D365 (3 dígitos ↔ 6 dígitos), reducir la carga operativa de administración de proveedores y asegurar que el código que fluye por todo el proceso (incluyendo la Declaración de Importación) sea el código D365 de 6 dígitos.

---

## Contexto del Proceso (As-Is)

1. En el SII actual existe una **tabla de exportadores** administrada manualmente, con código de 3 dígitos. Ejemplo: Caterpillar = `021`.
2. D365 usa un **Vendor Account de 6 dígitos**. Ejemplo: Caterpillar = `221372`.
3. Para conectar ambos sistemas existe una **tabla de mapeo** (3 dígitos ↔ 6 dígitos) que se mantiene manualmente: cuando llega un proveedor nuevo de D365, alguien debe registrarlo en la tabla del SII y crear la equivalencia.
4. Si el proveedor no existe en el SII, la factura no puede sincronizarse ni procesarse correctamente.
5. Los términos **"proveedor"** y **"exportador"** se usan indistintamente en el proceso actual, generando ambigüedad entre áreas.
6. La administración de la tabla de proveedores requiere intervención manual por parte de un usuario con permisos especiales (Cheche/TI o el equipo de Comercio Exterior).

---

## Descripción Funcional

El sistema debe:

1. **Al sincronizar facturas desde D365**, verificar si el código de proveedor D365 (6 dígitos) ya existe en el SII 2.0.
2. **Si el proveedor no existe**: crearlo automáticamente con los **datos mínimos** disponibles en D365 (código D365, nombre, país, estado activo), dejándolo disponible para complementación posterior.
3. **Permitir que el analista consulte y modifique** los datos del proveedor directamente en el SII 2.0, una vez creado: NIT, dirección, teléfono, y otros campos que se definan en sesión de detalle técnico.
4. **La sincronización de datos del proveedor es unidireccional**: viene de D365 → SII. Si el analista modifica datos en el SII, esos cambios **no se sincronizan de vuelta a D365**.
5. **Unificar la terminología**: usar solo el término **"Proveedor"** en toda la interfaz del SII 2.0, eliminando la ambigüedad con "Exportador".
6. **Eliminar la tabla de mapeo** SII (3 dígitos) ↔ D365 (6 dígitos): la tabla de exportadores de 3 dígitos queda **obsoleta**. Todos los registros nuevos usarán exclusivamente el código D365.

## Criterios de Aceptación (Gherkin)

### Escenario 1 – Proveedor ya existe en SII
```gherkin
Dado que llega una factura de D365 con código proveedor 221372 (Caterpillar)
Cuando se ejecute la sincronización
Y el código 221372 ya existe en el catálogo de proveedores del SII
Entonces el sistema debe asociar la factura al proveedor existente
  Y no debe crear un registro duplicado.
```

### Escenario 2 – Proveedor nuevo: creación automática con datos mínimos
```gherkin
Dado que llega una factura de D365 con un código proveedor que no existe en el SII 2.0
Cuando se ejecute la sincronización
Entonces el sistema debe crear automáticamente el proveedor con los datos mínimos de D365
  Y continuar el proceso de sincronización de la factura sin interrumpirlo
  Y dejar el proveedor en estado "Creado automáticamente - datos incompletos".
```

### Escenario 3 – Complementación de datos del proveedor
```gherkin
Dado que un proveedor fue creado automáticamente con datos mínimos
Cuando el analista acceda al módulo de gestión de proveedores
Entonces el sistema debe permitirle completar campos adicionales (NIT, dirección, teléfono)
  Y registrar la modificación con usuario, fecha y motivo.
```

### Escenario 4 – Sincronización unidireccional
```gherkin
Dado que el analista modificó el nombre de un proveedor directamente en el SII 2.0
Cuando D365 envíe la misma información en la próxima sincronización
Entonces el sistema NO debe sobrescribir los cambios realizados en el SII
  A menos que exista una política explícita de sincronización definida por el equipo técnico.
```

### Escenario 5 – Tabla de 3 dígitos queda inhabilitada
```gherkin
Dado que el SII 2.0 está en producción
Cuando un proceso intente usar la tabla de mapeo de 3 dígitos
Entonces el sistema debe ignorar dicha tabla
  Y usar exclusivamente el código D365 de 6 dígitos para identificar proveedores.
```

### Escenario 6 – Proveedor con actualizaciones en D365
```gherkin
Dado que un proveedor cambió su domicilio en D365
Cuando el SII ejecute el proceso de actualización de proveedores
Entonces el sistema debe reflejar el cambio en el SII 2.0
  Y registrar la actualización en el historial del proveedor.
```

## Reglas de Negocio

| ID    | Regla |
|-------|-------|
| RN-01 | El **código D365 de 6 dígitos (Vendor Account)** es el identificador oficial de proveedor en SII 2.0. No se usará ningún código de 3 dígitos. |
| RN-02 | Si un proveedor no existe en SII 2.0 al sincronizar una factura, se debe **crear automáticamente** con datos mínimos (código D365, nombre, país, estado activo). |
| RN-03 | La creación automática no interrumpe el proceso de sincronización de la factura. |
| RN-04 | Los datos del proveedor se sincronizan **únicamente de D365 → SII**. Cambios en el SII no se propagan a D365 de forma automática. |
| RN-05 | Los datos complementarios del proveedor (NIT, dirección, teléfono) deben poder editarse directamente en el SII con trazabilidad (usuario, fecha, motivo). |
| RN-06 | La terminología oficial en SII 2.0 es **"Proveedor"**. El término "Exportador" queda en desuso. |
| RN-07 | La tabla de mapeo SII (3 dígitos) ↔ D365 (6 dígitos) **queda obsoleta** con la entrada en vigor de SII 2.0. |
| RN-08 | Un proveedor creado automáticamente debe quedar marcado en estado **"Datos incompletos"** hasta que un analista revise y valide su información. |

## Notas Técnicas

- El código D365 es **bidireccional en el proceso**: entra al SII desde D365 y luego sale hacia la Declaración de Importación (DI) transmitida a la DIAN. Esto fue confirmado por Cheche (speaker_4) en sesión.
- La tabla de exportadores de 3 dígitos es un elemento heredado del sistema S400/SII v1. Debe establecerse una estrategia de migración para los registros históricos.
- La sincronización de proveedores ocurre en el momento de sincronizar la primera factura que los referencia, no en un proceso separado.
- El proceso de actualización ante cambios en D365 (ej: cambio de domicilio del proveedor) debe definirse como un proceso periódico o disparado por evento.
