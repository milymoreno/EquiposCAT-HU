# HU-IE-15-02: Consulta y Validación de Facturas Sincronizadas desde D365

**Proceso:** IMPORTACIÓN DE EQUIPOS CAT  
**Subproceso:** VALIDACIÓN DE FACTURAS  
**Requerimiento Original:** REQ-15 — Se requiere una validación previa por medio de un reporte que permita conciliar que lo registrado en D365 cayó correctamente al SII, por factura: CÓDIGO Y NOMBRE DEL PROVEEDOR – NÚMERO Y FECHA DE FACTURA – VALORES DE LA FACTURA – ID – OC – INCOTERM.  
**Versión:** 2.0  
**Fecha de revisión:** 2026-03-05  
**Estado:** Revisada con base en sesión de levantamiento (02-Mar-2026)

---

## Historia de Usuario

### Como:
Analista de Comercio Exterior

### Quiero:
Consultar y validar directamente en el SII las facturas sincronizadas desde D365, con capacidad de filtrar, detectar errores y corregir datos menores sin necesidad de salir del sistema.

### Para:
Confirmar que la información registrada en D365 cayó correctamente al SII antes de generar embarques, y evitar que errores en las facturas (como INCOTERM faltante) bloqueen el proceso posterior.

---

## Contexto del Proceso (As-Is)

1. Actualmente, el analista recibe por **correo electrónico** un archivo **Excel** generado por el SII con la información de las facturas sincronizadas.
2. El analista revisa el Excel manualmente para detectar inconsistencias entre D365 y el SII.
3. Si hay errores (ej: factura sin INCOTERM), el analista debe corregirlos antes de generar la guía de embarque.
4. El sistema actual (SII v1) permite asignar el INCOTERM manualmente a una factura que llegó sin él.
5. Si el error se detecta después de generar la guía, el analista debe **borrar la guía** y volver a empezar el proceso desde el paso de selección de facturas.
6. Una vez consolidados los costos (último paso del proceso), **no es posible corregir nada**: el ciclo queda cerrado.
7. La validación previa evita retrocesos costosos; hoy vive en una **bitácora manual en Excel** que el analista mantiene por su cuenta.

---

## Descripción Funcional

El sistema debe:

1. **Mostrar un listado de facturas sincronizadas** con todos sus campos: compañía, código y nombre del proveedor, número de factura, fecha, valor total, INCOTERM, moneda, ID equipo, OC y valores de cargos.
2. **Filtrar el listado** por los criterios definidos en la sección de filtros.
3. **Mostrar el estado de cada factura**: Sincronizada, Con error, Duplicada, Pendiente validación, Lista para embarque.
4. **Permitir corregir datos menores desde el SII**, sin volver a D365:
   - Asignar INCOTERM a facturas que llegaron sin él.
   - Cualquier campo que el sistema permita corregir debe quedar registrado con auditoría (usuario, fecha, motivo).
5. **Bloquear el avance al proceso de embarque** si la factura tiene estado "Con error" o "Pendiente validación".
6. **Exportar el resultado** a Excel, incluyendo solo los registros filtrados, con fecha/hora y usuario que genera el reporte.
7. **Reemplazar la bitácora manual en Excel** que hoy usan los analistas: esta funcionalidad es la versión digital, trazable y dentro del SII de ese proceso de conciliación.

## Criterios de Aceptación (Gherkin)

### Escenario 1 – Consulta general de facturas
```gherkin
Dado que existen facturas sincronizadas en el SII
Cuando el analista ingrese al módulo de consulta de facturas

  Y los filtros disponibles deben estar activos para su uso.
```

### Escenario 2 – Filtro por proveedor y fecha
```gherkin
Dado que el analista necesita ver facturas de un proveedor específico en un rango de fechas

```

### Escenario 3 – Exportación a Excel
```gherkin
Dado que el analista aplicó filtros y desea descargar la información
Cuando haga clic en exportar
Entonces el sistema debe generar un archivo Excel con solo los registros filtrados
  Y el archivo debe incluir fecha, hora y nombre del usuario que generó el reporte.
```

### Escenario 4 – Factura con error: bloqueo de embarque
```gherkin
Dado que una factura tiene estado "Con error" o "Pendiente validación"
Cuando el analista intente usarla para generar una guía de embarque

  Y mostrar un mensaje indicando el motivo del bloqueo.
```

### Escenario 5 – Corrección de INCOTERM desde el SII
```gherkin
Dado que una factura sincronizada llegó sin INCOTERM
Cuando el analista lo detecte en el módulo de consulta
Entonces el sistema debe permitirle asignar el INCOTERM directamente en el SII
  Y registrar la corrección con usuario, fecha y motivo.
```

### Escenario 6 – Visualización de detalle de error técnico
```gherkin
Dado que una factura tiene estado "Con error"
Cuando el analista la consulte

```

## Reglas de Negocio

| ID    | Regla |
|-------|-------|
| RN-01 | La consulta es de **solo lectura**, excepto para campos habilitados explícitamente para corrección menor (ej: INCOTERM faltante). |
| RN-02 | No se puede generar un embarque a partir de una factura con estado "Con error" o "Pendiente validación". |
| RN-03 | La corrección de datos menores en el SII debe quedar registrada con trazabilidad: usuario, fecha y motivo. |
| RN-04 | El archivo exportado a Excel debe incluir solo los registros que pasaron por el filtro, nunca el total sin filtrar. |
| RN-05 | Si una factura ya está asociada a una guía generada, no se puede modificar ni eliminar sin anular primero la guía. |
| RN-06 | Una factura solo puede avanzar al proceso de embarque si tiene estado "Lista para embarque". |
| RN-07 | La funcionalidad reemplaza la bitácora manual en Excel; debe ser suficientemente completa para que el analista no necesite mantener registros externos. |

---

## Estados de Factura

| Estado | Descripción |
|--------|-------------|
| Sincronizada | Llegó correctamente desde D365, sin errores detectados. |
| Con error | Falló alguna validación al sincronizarse; no puede usarse para embarque. |
| Duplicada | Llegó con misma combinación número + fecha que otra ya existente; no se procesó. |
| Pendiente validación | Requiere revisión manual antes de continuar el proceso. |
| Lista para embarque | Validada exitosamente; puede incluirse en una guía de embarque. |

## Notas Técnicas

- Actualmente, la validación se realiza sobre un Excel que llega por correo. En SII 2.0 debe estar completamente dentro del sistema con auditoría.
- El INCOTERM es un dato crítico: si falla la sincronización de este campo, el sistema actual lo permite corregir manualmente en el SII. Esta funcionalidad debe preservarse.
- Si un error se detecta después de generar la guía, el proceso de reversa implica borrar la guía y reiniciar. El objetivo de esta HU es **evitar ese escenario** con una validación previa robusta.
- La restricción de no modificar una factura ya asociada a una guía aplica para evitar inconsistencias entre la guía y las facturas que la componen.
